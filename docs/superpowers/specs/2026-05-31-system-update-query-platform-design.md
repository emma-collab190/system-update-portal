# 系統更新查詢平台 — 設計規格

**日期**：2026-05-31  
**版本**：v1.3（最終版，規格凍結）  
**目標**：讓導入夥伴與跨部門同仁能自助查詢系統更新記錄與邏輯，減少技術人員重複被問相同問題。

---

## 背景與問題

目前狀況：
- 系統更新資料存在 Google Sheets，由導入課從 Redmine 手動匯出維護（每月更新）
- 現有 HTML 工具需要每次手動複製貼上資料，重新整理後資料消失
- Claude API Key 直接暴露在前端（`F12` 即可看到）
- 沒有固定網址，需要手動傳檔案給使用者
- 技術人員持續被問四類重複問題：
  1. 最近系統有改什麼？
  2. 這個功能現在的邏輯是什麼？
  3. 這個 Bug 修了嗎？
  4. 這個更新會影響我的業務流程嗎？

---

## 目標使用者

- **導入夥伴**：正在導入系統的外部合作夥伴，需要了解系統最新邏輯
- **跨部門同仁**：業務部、產品設計部等非技術部門，需要了解系統更新對自身工作的影響
- **導入課**：資料維護者，每月更新 Google Sheets

---

## 整體架構

```
Redmine
   ↓ 手動匯出（維持現有流程）
Google Sheets（導入課每月更新）
   ↓ SpreadsheetApp 自動讀取
Google Apps Script Web App
   ├── GET  ?action=data    → 回傳所有更新記錄（JSON）
   ├── GET  ?action=search  → 關鍵字全文搜尋，回傳相關記錄
   └── POST ?action=ask     → 先關鍵字過濾 → 呼叫 LLM → 回傳答案（含來源）
   ↓
前端 HTML（GitHub Pages，固定 URL）
   ├── AI 問答（附引用來源）
   ├── 全文搜尋
   ├── 最近更新
   └── 熱門問題
```

## Google Sheets 資料欄位

| 欄位 | 說明 | 範例 |
|---|---|---|
| 日期 | 正式機更新日期 | 2026/5/22 |
| 系統別 | FonTour / FonTicket / Traveline | FonTicket |
| 摘要 | 簡短說明（20 字內），見「摘要維護策略」 | 新增票券效期設定 |
| 更新內容 | 完整說明（原有欄位） | 針對效期模式與核銷設定值欄位... |
| Redmine 編號 | 任務編號 | 31664 |
| Redmine 連結 | 任務連結 | https://redmine.example.com/... |
| 負責單位 | 研發處 | 研發處 |
| 更新類型 | 產品線功能優化 / BUG 修復 / 程式邏輯調整... | 產品線功能優化 |
| 相關部門 | 此更新影響哪些部門 | 業務部 |
| **影響程度** | **新增**：高 / 中 / 低 | 高 |

### 影響程度定義

| 等級 | 說明 | 範例 |
|---|---|---|
| 高 | 業務流程或對外功能改變，相關人員需要知道 | 退款流程改版、新增必填欄位 |
| 中 | 部分操作調整，有影響但不緊急 | 介面改版、欄位名稱調整 |
| 低 | 內部優化或 Bug 修復，使用者通常無感 | 效能改善、錯誤訊息修正 |

---

## 摘要維護策略

摘要欄位是前端卡片預覽與 AI 問答的核心依據，但人工維護容易隨時間品質下降。採用以下方式確保品質：

**正常流程（方案 A，推薦）：**
1. 導入課更新 Google Sheets 後，按「產生摘要」按鈕
2. Apps Script 呼叫 LLM，根據「更新內容」自動生成 20 字以內摘要
3. 導入課確認或微調後儲存

**備援流程（方案 B）：**  
若摘要欄位空白，前端與 AI 自動取「更新內容」前 60 字作為摘要，不強制人工維護。

## Apps Script 後端設計

### 端點

**`GET ?action=data`**
- 讀取 Google Sheets 所有資料列
- 使用 `CacheService` 快取 30 分鐘
- 回傳 JSON：

```json
{
  "updatedAt": "2026-05-31 09:00",
  "records": [
    {
      "date": "2026/5/22",
      "system": "FonTicket",
      "summary": "新增票券效期設定",
      "content": "針對效期模式與核銷設定值欄位...",
      "redmineNo": "31664",
      "redmineUrl": "https://redmine.example.com/issues/31664",
      "dept": "研發處",
      "type": "產品線功能優化",
      "relDept": "研發處",
      "impact": "中"
    }
  ]
}
```

**`GET ?action=search&q=退款`**
- 對 `summary`、`content` 欄位做關鍵字比對
- 回傳最相關的前 20 筆，格式同上

**`POST ?action=ask`**
- 接收：`{ "question": "退款流程是什麼？" }`
- 先在 Apps Script 側做關鍵字過濾，從問題中擷取關鍵詞，找出相關記錄（最多 20 筆）
- 只將這 20 筆送給 LLM（不是全部資料）
- 呼叫 `claude-haiku-4-5-20251001`，max_tokens: 1000
- 回傳：`{ "answer": "...", "sources": [...] }`

**關鍵字過濾範例：**
```
使用者提問：退款流程是什麼？
↓ 擷取關鍵詞：退款、退票、退費
↓ 比對 summary + content
↓ 找到 #31420、#31801、#32250
↓ 只送這 3 筆給 LLM
```

好處：資料量增長到 3000+ 筆也不會 Context 爆掉，回答更準確，Token 成本更低。

**摘要自動生成（Google Sheets 自訂選單，非 Web API）**
- 導入課在 Sheets 中透過「自訂選單 → 產生摘要」觸發
- Apps Script 直接讀取選取列的「更新內容」，呼叫 LLM，寫回「摘要」欄位
- 不需要額外 Web API 端點，架構更單純
- 導入課確認或微調後儲存即完成

### AI 系統提示詞

```
【最高優先規則】
你只能根據下方提供的更新紀錄回答問題。
若資料不足，必須直接說：「目前更新資料中沒有足夠資訊回答此問題」
不得使用任何外部知識。不得推測。不得補充資料中未出現的資訊。
寧可說不知道，也不能一本正經地講出錯誤資訊。

你是「系統更新查詢助理」，服務對象是業務部、產品設計部等跨部門同仁與導入夥伴。

系統別說明：
- FonTour：套票系統
- FonTicket：票務系統
- Traveline：前端網頁

回答格式規則：
1. 條列式呈現，每筆說明重點影響
2. 每筆必須附上：日期、Redmine 編號、系統別、影響程度、業務影響
3. 若跟業務推廣或產品設計有關，特別說明「業務影響」
4. 語氣友善，避免技術術語
```

### AI 回答格式（含引用來源）

```
✦ 退款狀態改為人工確認

   日期：2026/05/22
   來源：#31420
   系統：FonTicket
   影響程度：高
   業務影響：超過退款期限的訂單改由人工審核，
            業務同仁需確認退款申請流程是否需調整。
```

### 熱門主題追蹤

- Apps Script 每次收到問題時，從問題中擷取關鍵主題詞（如「退款」「折價券」「核銷」），寫入 Sheets 的「查詢記錄」分頁
- 統計主題詞頻率，而非完整問題（避免「退款怎麼處理」「退款流程是什麼」被計為不同項）
- 前端載入時取出現次數最多的前 5 個主題，顯示為「本月熱門主題」供一鍵點擊

### 熱門更新追蹤

- Apps Script 每次收到 `/search` 或 `/ask` 請求命中某筆記錄時，累計該記錄的查詢次數
- 記錄格式：`{ "redmineNo": "31664", "viewCount": 32, "lastViewed": "2026-05-31" }`
- 前端可顯示「本月最多人查看」前 5 筆更新，比熱門問題更直接反映使用者關注點

## 前端設計

### Tab 結構

| Tab | 說明 |
|---|---|
| AI 問答 | 自然語言提問，AI 根據相關更新記錄回答（附引用來源） |
| 全文搜尋 | 直接搜尋關鍵字或 Redmine 編號，不依賴 AI |
| 最近更新 | 依時間排序，最新在前，標示本月新增 |

### 頁面頂部

- **首次使用說明**：
  > 「這裡記錄了 FonTour、FonTicket、Traveline 的系統更新歷程。你可以用 AI 問答直接提問，或在瀏覽更新中篩選查看。資料每月由導入課更新。」
- **資料狀態**：`資料最後更新：2026/05/30`

### 更新分類標籤對照

前端將「更新類型」轉換為易讀的彩色標籤：

| 原始更新類型 | 前端標籤 | 顏色 |
|---|---|---|
| 新開發 / 新增功能 | 🟢 新功能 | 綠色 |
| 產品線功能優化 / 流程調整 | 🔵 流程調整 | 藍色 |
| BUG 修復 | 🟡 Bug修復 | 黃色 |
| 程式邏輯調整（高影響） | 🔴 重大異動 | 紅色 |
| 其他 / 未分類 | ⬜ 內部調整 | 灰色 |

> 「重大異動」紅標的觸發條件：更新類型含「邏輯調整」**且**影響程度為「高」。

### AI 問答

快速問題 chips 依使用情境分組：

```
[了解最新]   這週有哪些新功能？    最近有什麼重大異動？
[查功能邏輯] 折價券目前怎麼運作？  退款流程是什麼？
[查 Bug]     這個問題修好了嗎？    最近修了哪些 Bug？
[業務影響]   哪些更新影響業務推廣？ 有哪些前台頁面改版？
```

**本月熱門問題**區塊（查詢頻率自動統計）：
```
本月熱門問題
1. 折價券邏輯
2. 退款流程
3. 核銷設定
```

### 全文搜尋（不依賴 AI）

- 搜尋框輸入關鍵字或 Redmine 編號
- 即時顯示匹配的更新記錄
- 卡片顯示：摘要、日期、系統別、影響程度標籤
- 點開卡片展開完整更新內容

### 更新卡片設計

```
┌─────────────────────────────────────────────────┐
│ 新增票券效期設定           FonTicket  🔵流程調整 │
│ 2026/5/22 · 影響程度：中                         │
│ 相關部門：業務部                                 │
│                                [展開全文] [#31664]│

```

點開後顯示完整 `content` 欄位與 Redmine 連結。

## 安全設計

| 項目 | 做法 |
|---|---|
| Claude API Key | 存於 Apps Script Script Properties，前端不可見 |
| 公司內部同仁 | Apps Script Web App 設定「限 Google Workspace 網域（公司信箱）」，登入才能存取 |
| 外部導入夥伴 | 建立 Google Group，授權特定帳號加入，Apps Script 設定允許此 Group 存取 |
| Sheets 資料 | Google Sheet 本身不公開，只有 Apps Script 有讀取權限 |
| 不使用通行碼 | `?password=xxx` 安全性不足，不採用 |

## 部署流程（一次性設定）

```
1. Google Sheets 準備
   a. 新增「摘要」欄位、「影響程度」欄位
   b. 導入課可用「產生摘要」功能批次補填摘要

2. 建立 Google Apps Script 專案
   a. 綁定現有 Google Sheets
   b. 將 Claude API Key 存入 Script Properties
   c. 部署為 Web App
      - 公司內部：限 Google Workspace 網域
      - 外部夥伴：授權 Google Group
   d. 記下 Web App URL

3. 前端 HTML 上傳至 GitHub Pages
   a. 將 Apps Script Web App URL 填入前端設定變數
   b. 發布固定網址給所有使用者

4. 日後維護
   └── 導入課更新 Google Sheet → 平台 30 分鐘內自動同步
```

## LLM 設計原則

Apps Script 中以設定變數管理 LLM Provider，不寫死模型 ID：

```javascript
const LLM_PROVIDER = "anthropic"; // 可切換：anthropic / openai / google
```

預設採用 `claude-haiku-4-5-20251001`（成本低，足以處理查詢整理場景）。  
換模型時只改設定變數，不需修改業務邏輯或前端。  
若未來呼叫頻繁，可評估 Gemini Flash 或 GPT-4o mini（查詢場景不需強推理能力）。

## 開發優先順序

| 優先 | 功能 | 理由 |
|---|---|---|
| P0 | Apps Script 後端（三個端點） | 架構基礎 |
| P0 | 前端改為從 Apps Script 拉資料 | 解決最大痛點 |
| P0 | Claude Key 移至後端 | 安全必做 |
| P1 | AI 問答（含關鍵字預過濾、引用來源、不猜測規則） | 核心功能 |
| P1 | 全文搜尋 | 與 AI 互補，覆蓋不想聊天的使用者 |
| P1 | 更新分類標籤（前端轉換） | 讓業務一眼看懂，無需後端改動 |
| P1 | 影響程度欄位（高/中/低） | 業務最在意「要不要在意這筆更新」 |
| P2 | 摘要自動生成（LLM 輔助） | 確保摘要長期品質 |
| P2 | 熱門問題追蹤 | 提升使用體驗 |
